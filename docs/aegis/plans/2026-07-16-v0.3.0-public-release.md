# Goal

Deliver CopperSkin 0.3.0.0 as the first official public release: a dependable, documented WPF skinning library with a usable graphical editor for icons and basic painting, broader standard-control coverage, reproducible packages, and a release process that can be repeated from a clean checkout.

The editor is intentionally a professional icon-and-light-paint authoring tool, not a general illustration suite. It must support vector-first icon work, simple freehand painting, layers, selection, transforms, undo/redo, zoom, save/reopen, and export. Full raster brush engines, filters, gradients, animation, and collaboration are outside this release.

# Architecture

CopperSkin keeps four ownership boundaries:

- CopperSkin.Core owns the renderer-neutral graphics document model, validation, serialization, schema versioning, and pack persistence. It must not reference WPF types.
- CopperSkin.Wpf owns the themed graphics renderer, hit testing, zoom/pan surface, icon display control, and resource dictionaries/templates for supported WPF controls. It consumes Core documents and the existing IThemeAwareDrawingSurface/DrawingThemeSnapshot seam.
- CopperSkin.Designer owns editor interaction state: tools, selection, commands, undo/redo, layers, palettes, import/export commands, and the authoring UI. It calls the WPF surface rather than duplicating rendering logic.
- CopperSkin.Cli remains a headless authoring/release surface. It may validate, inspect, and export graphics documents, but it must not become a second interactive editor or renderer.

The editor uses one vector-first GraphicDocument model for both icon documents and basic painting documents. Icon mode supplies transparent pixel-oriented presets and SVG/XAML/PNG export. Paint mode permits arbitrary canvas sizes, background color, freehand strokes, simple shape layers, and raster preview/export.

The WPF expansion is organized as resource-dictionary slices rather than one continually growing CopperSkin.Controls.xaml. A single public merge point preserves existing application startup behavior while internal dictionaries separate primitives/input, data/navigation, documents/media, menus/toolbars, and editor surfaces. The coverage matrix is the authority for what is supported, partial, or intentionally excluded.

## Requirement Ready Check

- Requirement source: the repository owner requests a v0.3.0.0 public release with a proper icon/basic-paint editor and substantially broader WPF standard-control replacement.
- Public-release acceptance: package metadata is 0.3.0.0, the documented API/schema is stable, clean-checkout CI passes, packages and release archives validate, and the README contains runnable editor/library examples.
- Graphics acceptance: create an icon and a paint document in the Designer, edit selection/layers/tools, undo/redo, save/reopen, apply the active theme, and export SVG/XAML/PNG with deterministic results.
- WPF acceptance: the coverage matrix identifies the concrete v0.3 target set; each new or changed style has keyboard/focus, validation, high-DPI, RTL, theme-switch, and automation expectations recorded and tested where applicable.
- Release decision: use 0.3.0.0 for assembly/file/informational/package versions. The release owner must approve the repository license before tagging; MIT is the recommended default for a permissively reusable WPF library, but the committed license file is authoritative.

## Compatibility Boundary

- Preserve existing public namespaces, target frameworks, theme-token keys, signed theme-pack readability, and corrected horizontal-scroll behavior.
- Add graphics APIs under a new CopperSkin.Core.Graphics namespace and keep them WPF-free. Do not expose System.Windows.Media, BitmapSource, or dispatcher-affine objects from Core.
- Version the graphics document schema separately from the assembly version. Readers reject unsupported future schema versions with a clear diagnostic and round-trip every v0.3 element without lossy conversion.
- Existing theme packs without a graphics section remain valid. Graphics-enabled packs remain readable by Core-only validation tools.
- New WPF styles must not change application behavior outside an opted-in CopperSkin resource scope. Existing CopperSkinThemeScope and explicit resource merges remain the activation boundary.
- Keep unsupported WPF surfaces explicit. WebBrowser, OS-hosted media, and externally owned visual trees must not be called fully themed merely because a style key exists.
- The editor is Windows/WPF-only, but Core document validation and serialization remain testable on every Core target.

# Plan-Time Complexity Check

This is a substantial feature release, not a Planless Slice. Complexity is bounded by one Core document model instead of separate icon and paint models; one WPF renderer shared by Designer and runtime; one command-based history; one resource merge point; a fixed tested control target list; and explicit exclusion of full raster editing, filters, animation, and external-host controls.

# Architecture Integrity Lens

The following failures are disallowed:

- WPF types leaking into CopperSkin.Core.
- Designer rendering diverging from runtime WPF rendering.
- Graphics documents being opaque Designer state with no Core validator or CLI inspection path.
- New control templates silently replacing system defaults outside CopperSkinThemeScope.
- A control being marked supported because its style parses while focus, keyboard, automation, validation, or high-DPI behavior is broken.
- Version/schema changes being scattered across project files without one release contract.
- Tests that only assert resource existence without exercising document or control behavior.

# Change Necessity and Existence Check

The graphics model, renderer, editor command layer, export pipeline, coverage inventory, and public-release metadata are necessary new capabilities; no equivalent implementation exists in the tree. Existing theme/token editing, drawing-surface abstractions, control dictionaries, pack serializers, and deployment documentation must be extended rather than replaced. Before creating an abstraction, search for the corresponding existing type and add behavior to that owner when one exists.

# Risks

- Graphics scope expansion: keep the v0.3 element/tool list fixed and move filters, gradients, raster brushes, animation, and collaboration to a later roadmap.
- Model/renderer drift: keep Core as serialized authority and require renderer/export golden tests for every element kind.
- WPF template fragility: DataGrid, DatePicker, RichTextBox, MediaElement, and popup-heavy surfaces vary by platform. Use behavior tests and document partial support instead of forcing a full-support claim.
- Performance: large freehand paths or many layers can make invalidation expensive. Start with dirty-region invalidation and a documented v0.3 document-size guideline; profile before adding caching complexity.
- Schema migration: keep schema versioning independent, reject future versions clearly, and add an upgrade path only when a migration is specified.
- Release governance: missing license approval, package feed, signing key, or protected tag is an external blocker. Record the owner and decision in the release checklist before tagging.
- Accessibility/regression debt: treat focus and automation checks as acceptance criteria, not polish work.
