# ADR-0001: Keep graphics documents renderer-neutral

- Status: Accepted
- Date: 2026-07-19
- Scope: CopperSkin 0.3 graphics editor and runtime surfaces

## Context

CopperSkin needs one persisted model for icons and basic painting that can be authored in the Designer, validated by the CLI, embedded in theme packs, rendered at runtime, and exported without making the Core library depend on WPF. Maintaining separate editor and runtime models would create drift in serialization, hit testing, and export behavior.

## Decision

`CopperSkin.Core.Graphics` owns the versioned `GraphicDocument`, layers, elements, geometry, styles, transforms, validation, and deterministic JSON serialization. It contains no WPF types.

`CopperSkin.Wpf` owns `GraphicCanvas`, `CopperIcon`, hit testing, theme-token brush resolution, and SVG/XAML/PNG adapters. `CopperSkin.Designer` owns tools, selection, layer commands, history, and the authoring UI. `CopperSkin.Cli` remains headless and is limited to validation, inspection, and renderer-free canonical export.

Graphics styles may refer to a CopperSkin color token. The active WPF `DrawingThemeSnapshot` resolves that token, while Core preserves the reference and its fallback color.

## Consequences

- A `.cgraphic` document can be validated and signed without loading WPF.
- Runtime and Designer rendering share one WPF implementation.
- Schema versioning is independent of the assembly/package version.
- WPF-only features such as rasterization and DPI conversion stay at the boundary.
- Full raster brushes, filters, gradients, animation, and collaboration remain outside v0.3.

## Alternatives considered

- Put WPF `Geometry`, `Brush`, and `BitmapSource` objects in the document model: rejected because it would break Core portability and headless tooling.
- Let the Designer own an opaque document format: rejected because runtime and CLI consumers need a stable contract.
- Implement separate icon and paint models: rejected because layers, styles, validation, and export would diverge.

## Verification

- Core graphics round-trip, schema, validation, and theme-pack tests pass.
- WPF renderer, hit-test, theme-token, extent, export, and visual tests pass on .NET 8, 9, and 10.
- CLI graphics validation/inspection/canonical export tests pass.
- Core and WPF packages contain README, LICENSE, target assets, and XML documentation.
